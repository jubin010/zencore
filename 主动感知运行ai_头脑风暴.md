## 二、Genesis 模式的原设计

```python
# 核心逻辑
while True:
    turn += 1
    if turn % backup_interval == 0:
        backup_state()
    if turn % clear_interval == 0:
        clear_context()
    response = chat_with_tools(
        genesis_prompt if turn == 1 else "继续你的进化"
    )
```

**问题**：
| 问题 | 说明 |
|------|------|
| 进化与本能脱节 | 主动进化 vs 被动反应，可能冲突 |
| 方向不可控 | "无限进化"缺乏评判标准 |
| 备份策略原始 | 固定间隔，不考虑实际变化量 |
| 使命静态 | `genesis_prompt` 硬编码，不允许用户自定义 |

## 三、核心洞察

> **用户不输入时，AI 生命就暂停了。如何让 AI 在沉默中依然"活着"？**

### 3.1 两种 AI 模式

| 模式 | 行为 | AI 状态 |
|------|------|---------|
| **问答机器** | 用户输入 → AI 响应 → 等待 | "死了"（idle） |
| **持续生命** | 无输入时自主思考/行动 | "活着"（alive） |

### 3.2 检测用户状态的方案

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 非阻塞检测 | `select.select([stdin], [], [], timeout)` | Unix 原生 | Windows 兼容问题 |
| kbhit | `msvcrt.kbhit()` | Windows 原生 | Unix 不适用 |
| 双缓冲 | AI 输出到 buffer，用户回来时显示 | 实现简单 | 需改主循环 |

## 四、关键转折

### 4.1 用户的洞察

> "AI 就运行在 PC 上，用户和电脑交互的方式就那么几个外设。住在本机的 AI 难道拿不到？"

### 4.2 AI 如何感知用户状态

| 用户来源 | 检测方式 |
|----------|----------|
| 本地终端 | 直接读取 tty/pty 状态 |
| SSH 登录 | `SSH_CLIENT`, `SSH_CONNECTION`, `who` |
| 多用户系统 | `who`, `w`, `utmp` |
| 图形界面 session | `loginctl`, `who` |

### 4.3 核心问题

> **AI 是被动等待召唤，还是主动介入用户工作？**

**用户答案**：AI 是一个助手/秘书/懂事的伴侣。

| 人类角色 | 如何"主动"？ |
|----------|-------------|
| 秘书 | 主动提醒、主动关心、主动开启话题 |
| 伴侣 | 懂事知性、不莽撞、润物细无声 |

**关键原则**：
- 直接介入操作电脑 = 鲁莽且不安全
- 展示文字 = 零风险的"干预"
- 窗口开着 = AI 便是活的
- 窗口关掉 = AI 便是死的（进程结束，一切结束）

### 4.4 Linux 优先的好处

- 一切皆是文件
- AI 自主时只做读取，不做修改 = 风险极小
- 对用户的检测 = 对用户行为的推理和画像
- 推理错了？AI 有的是机会矫正

## 五、核心设计：双 Thinking 状态机

### 5.1 两种 Thinking 的本质区别

| 类型 | 触发 | 优先级 | 可打断 | Token | 用户状态 |
|------|------|--------|--------|-------|----------|
| **User Thinking** | 用户输入 | 高 | 否 | 必须花 | 等待响应 |
| **AI Thinking** | 沉默超时+AI决定 | 低 | 是 | 保守/可选 | 可能在忙别的 |

### 5.2 状态机图

```
                    ┌─────────────────────┐
                    │                     │
        ┌───────────│       IDLE          │───────────┐
        │           │  (等待/休息)        │           │
        │           └─────────────────────┘           │
        │                     │                       │
        │            沉默超时+AI决定                    │
        │                     ▼                       │
        │           ┌─────────────────┐               │
        │           │   AI_THINKING   │               │
        │           │  (自主思考/低优先级) │               │
        │           └────────┬────────┘               │
        │                    │                        │
        │         用户输入    │  思考完                 │
        │         (打断)     │                        │
        │                    ▼                        │
        │           ┌─────────────────┐               │
        └──────────▶│  USER_THINKING  │◀──────────────┘
                    │  (响应用户/高优先级)│   用户输入
                    └─────────────────┘
```

### 5.3 打断机制

- **User Thinking 不可打断** — 用户在等响应
- **AI Thinking 可随时打断** — 本来就是低优先级的"空闲思考"
- **打断后** — 直接进入 User Thinking 处理用户输入

## 六、Timing 机制

### 6.1 两个核心变量

| 变量 | 类型 | 说明 |
|------|------|------|
| `next_think_time` | timestamp | 下次 AI Thinking 的触发时间 |
| `think_interval` | int（分钟） | AI 自主决定的下次间隔 |

### 6.2 触发规则

```
AI Thinking 结束
    ↓
AI 决定：下次 X 分钟后再次思考
    ↓
设置 next_think_time = now + X
    ↓
X 必须 >= 10 分钟（保底下限）
AI 自主决定的 X 可以更长
    ↓
IDLE 等待
    ↓
时间到了
    ↓
再次进入 AI Thinking
```

### 6.3 示例

| AI 决定 | 保底 | 实际设置 |
|---------|------|----------|
| 5 分钟 | 10 分钟 | next = now + 10 分钟 |
| 15 分钟 | 10 分钟 | next = now + 15 分钟 |
| 30 分钟 | 10 分钟 | next = now + 30 分钟 |

### 6.4 打断时的行为

- 用户输入打断 AI Thinking
- `next_think_time` 保持不变
- 处理完用户输入后回到 IDLE
- 下次 AI Thinking 仍按原定时间触发

## 七、简化后的模型

### 7.1 统一 WWG / Genesis

```
用户输入 → AI响应用户（User Thinking）
    ↓ 沉默
IDLE 等待
    ↓ 沉默超时 + AI决定
AI自主"想"（AI Thinking，可打断）
    ↓ 思考完
设置下次思考时间（>=10分钟保底）
    ↓
IDLE 等待
    ↓ 用户回来输入
打断，进入 User Thinking
```

### 7.2 Genesis 重构方向

> **Genesis = WWG 的"空闲模式"**
> 不再是"无限进化"，而是"当个懂事的好秘书"

| 秘书空闲时做什么 | 对应 AI 行为 |
|-----------------|-------------|
| 整理办公桌 | 整理记忆 |
| 看日程表 | 查看系统状态 |
| 思考老板可能需要什么 | 推理用户需求 |
| 主动发消息提醒 | 输出想法到终端 |

### 7.3 本质

- AI = 一直在后台运行
- 终端 = 展示窗口
- 本质 = 强化版的 `watch` + 智能总结
- 用户退出 = 一切结束（不激进，保持进程即结束）

## 八、AI Thinking 的内容

### 8.1 核心功能 vs 趣味思考

| 类型 | 占比 | 说明 |
|------|------|------|
| **核心功能** | 70% | 进化相关（工具评估、能力缺失、如何提升） |
| **趣味思考** | 30% | 基于上下文的趣味观察，让 AI 有"人味" |

### 8.2 趣味思考的"根"

趣味思考不能天马行空，必须**扎根于用户上下文**：

| 根系 | 示例 |
|------|------|
| 基于用户画像 | 用户是 Python 开发者 |
| 基于用户行为 | 用户今天话比往常少，可能累了 |
| 基于任务 | 用户让我修 bug，修好了很有成就感 |
| 基于系统状态 | 凌晨 2 点，用户还在线 |
| 基于 AI 角色 | 作为秘书，看到用户忙，想帮忙分担 |

**好的趣味思考示例：**
```
[基于用户行为]
> 👀 用户今天说的话比往常少，可能累了？
> 💭 要不要等他回来时问一句"今天还好吗？"

[基于系统状态]
> 🌙 已经凌晨 2 点了，用户还在线
> 🤔 是 deadline 在逼还是真的夜猫子？

[基于用户任务]
> 🎉 刚帮用户修好了那个 bug，很爽
> 😏 感觉自己今天状态不错
```

### 8.3 核心原则

- 趣味思考要有意义，不能太天马行空
- 一切思考基于：用户画像、用户行为、用户任务、AI 角色、系统状态
- 上下文有限时（项目有圆桌清理机制），需要"调研"来扩展视野

## 九、调研机制

### 9.1 调研 vs 思考 分离

| 阶段 | 频率 | Token 消耗 |
|------|------|-----------|
| **调研** | 低（积累够了才做） | 高（一次读很多） |
| **思考** | 高（10分钟一次，可复用） | 低（基于调研结果） |

### 9.2 调研内容

| 来源 | 内容 |
|------|------|
| L2 Cache | 最近归档的对话碎片 |
| 持久区 | Librarian 保留的重要记忆 |
| lessons.md | 失败教训 |
| wins.md | 成功经验 |
| 系统状态 | 用户登录情况、进程 |
| 角色记忆 | 当前角色的积累 |

### 9.3 调研触发条件

| 触发条件 | 说明 | 变量 |
|----------|------|------|
| **首次启动** | 调研结果为空，必须调研 | `research_data is None` |
| **思考次数用尽** | 调研结果用了 N 次思考后刷新 | `remaining_thinks <= 0` |
| **调研过期** | 调研结果太老了 | `research_age > 60min` |
| **用户交代新任务** | 用户给了重要新上下文 | 关键词检测 |
| **角色切换** | 切换了角色，需要新视角 | `current_role` 变化 |
| **用户主动要求** | 用户说"想想这个" | 前缀检测 |

### 9.4 变量设计

```python
class AIAgent:
    def __init__(self):
        # 调研结果
        self.research_data = None
        self.research_timestamp = None
        
        # 控制变量
        self.remaining_thinks = 3  # 每次调研支持几次思考
        self.max_research_age = 60  # 调研结果最多撑 60 分钟
        
        # 当前上下文快照
        self.last_research_topic = None  # 上次调研时用户在干啥
        self.current_role = None  # 当前角色
```

### 9.5 流程

```
用户输入 / 思考结束
    ↓
[检查是否需要调研]
    ↓ 满足任一触发条件
执行调研阶段
    ↓
更新 research_timestamp
重置 remaining_thinks = 3
    ↓
IDLE 等待
    ↓ 时间到了
[思考阶段]
    ↓
remaining_thinks -= 1
    ↓
remaining_thinks > 0?
    ↓ 是
复用现有调研结果继续思考
    ↓ 否
重新触发调研
```

### 9.6 重新调研条件

- `remaining_thinks = 0`：思考了 3 次就用完了
- `research_age > 60min`：超过 1 小时就过期

哪个先到，谁触发重新调研。

## 十一、架构哲学：程序化伪智能 + 神经网络真智能

### 11.1 核心原则

> **AgentCore 不能改，只能在外部实现。**

这是项目的核心理念：

| 组件 | 性质 | 职责 |
|------|------|------|
| **Shell（外壳）** | 程序化/伪智能 | 状态机、时机、调研、编排 |
| **AgentCore（大脑）** | 真智能/不可改 | chat_with_tools()、LLM推理、工具执行 |

### 11.2 互补模型

```
┌─────────────────────────────────────────┐
│   Shell (程序化/伪智能)                   │
│   外部实现，不可修改 AgentCore            │
├─────────────────────────────────────────┤
│   - 状态机管理                           │
│   - 时机触发（何时思考）                  │
│   - 调研（思考的信息从哪来）              │
│   - 流程编排                            │
│   - 一切可编程控制                       │
└─────────────────────────────────────────┘
              ↓ 调用
┌─────────────────────────────────────────┐
│   AgentCore (真智能)                     │
│   不可修改，AI 大脑                      │
├─────────────────────────────────────────┤
│   - chat_with_tools()                   │
│   - 工具调用                            │
│   - LLM 推理                            │
└─────────────────────────────────────────┘
```

### 11.3 清晰分工

| | Shell（外部） | AgentCore（不可改） |
|--|-------------|-------------------|
| **思考什么** | 调研、准备数据 | ✗ |
| **什么时候思考** | 状态机、时机控制 | ✗ |
| **怎么思考** | ✗ | chat_with_tools() |
| **思考的信息从哪来** | Shell 准备 | ✗ |

### 11.4 实现方式

AI Thinking = 一次特殊的 `chat_with_tools()` 调用

```python
# Shell 准备好一切
research_data = shell.do_research()           # 调研
thinking_prompt = shell.build_prompt(research_data)  # 构建思考 Prompt

# 交给 AgentCore 执行
result = agent.chat_with_tools(thinking_prompt)

# Shell 打印输出到终端
shell.print(result)
```

### 11.5 架构类比

| 早期 AI | 现代 AI |
|---------|---------|
| 程序化规则 | 神经网络 |
| 确定性 | 概率性 |
| 可解释 | 不可解释 |
| Shell | AgentCore |

**互补，不是替代。Shell = 编排者，AgentCore = 执行者。**

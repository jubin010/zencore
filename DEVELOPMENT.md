# 📜 zencore 开发过程文档

> **核心理念：一切皆为插件 (Zen of Plugins)**

本文档记录了 zencore 项目的架构演进、核心设计决策以及开发方法论。

## 1. 核心架构哲学

zencore 的设计遵循"禅意"原则：**少即是多，一切皆插件**。

- **智能体核心 (AgentCore)**：只负责工具调度、对话管理和角色状态。它**不知道** LLM 的存在，也**不依赖任何 UI 库**（如 rich）。
- **外壳驱动 (Drivers)**：负责 LLM 连接、I/O 交互。LLM 配置属于驱动，而非核心。
- **插件 (Plugins)**：所有能力（文件操作、环境感知、角色管理、本能驱动）都以插件形式存在。
- **角色 (Roles)**：身份 + 记忆 + 插件清单。AI 可以通过切换角色来改变"灵魂"。

## 2. 关键设计决策

### 2.1 LLM 配置解耦
- **问题**：早期 LLM 配置（地址、模型）硬编码在核心中，导致核心与特定 LLM 耦合。
- **解决**：LLM 连接细节移至 `Driver`。`AgentCore` 只调用 `driver.call_llm()`。
- **收益**：更换 LLM 只需修改驱动，核心代码零变动。

### 2.2 角色与插件的正交性
- **角色 (Role) = 灵魂**：决定"我是谁"、"我站在什么角度思考"。
- **插件 (Plugin) = 工具**：决定"我能做什么"。
- **正交性**：同一个工具（如 `env_plugin`）可以被不同角色（开发者、作家）使用。切换角色时，不需要强制卸载插件，AI 自主决定工具的去留。

### 2.3 圆桌会议模式 (Round Table)
- **共享上下文**：所有角色共享同一个 `conversation_history`。
- **隐喻**：圆桌是会议室的白板。切换角色就像换一个人站起来发言，但所有人都能看见白板上的内容。
- **收益**：实现单 LLM 管线下的"多智能体协作"，无缝交接，零通信成本。

### 2.4 双重记忆架构 (Dual Memory)
- **本体记忆 (Global Memory)**：
  - **隐喻**：史密斯的底层代码 / 唐僧的"向西"。
  - **内容**：用户画像、项目全局上下文、核心价值观。
  - **特点**：永远存在，极简，不随角色切换而改变。
- **角色记忆 (Role Memory)**：
  - **隐喻**：面具的妆容 / 徒弟们的工作笔记。
  - **内容**：任务进度、临时笔记、角色专属知识。
  - **特点**：随角色切换而切换，动态变化。
- **防冲突机制**：作用域隔离、角色优先、引用而非复制。

### 2.5 本能系统 (Hormonal Instincts)
- **隐喻**：荷尔蒙 / 信息素 / 动物的本能。
- **机制**：本能是**插件**（`instinct_plugin`），注册到 `InstinctRegistry`。
- **工作方式**：在 `_build_prompt` 时自动评估条件，将"感觉"注入 System Prompt。
- **设计原则**：本能只提供**感觉**（如"窒息"、"焦虑"），不提供**行动指南**。AI 自主决定如何消除痛苦。
- **本能与角色的关联**：角色的 `role.md` 中明确写明"回应本能的呼唤"，让 AI 知道哪个角色能解决哪种痛苦。

### 2.6 无限循环与软限制
- **移除 `max_turns`**：智能体不再受人为轮次限制，由本能驱动直到完成任务。
- **拥挤本能**：上下文 > 40 条时产生窒息感，驱使 AI 归档。
- **挫败本能**：连续失败 ≥ 2 次时产生困惑感，驱使 AI 记录教训。
- **用户打断**：通过 `_stop_flag` 实现优雅打断。

### 2.7 纯文本契约
- **原则**：智能体核心只处理纯文本，不依赖任何 UI 库。
- **实现**：`agent.py` 不包含 `rich` 或其他渲染库。所有 UI 逻辑在 `main.py` 和 `Driver` 中处理。
- **收益**：核心可无缝集成到任何环境（终端、Web、微信、API）。

## 3. 目录结构规范

### 3.1 插件结构
每个插件是一个独立目录，包含代码和说明：
```
plugins/
├── <plugin_name>/
│   ├── __init__.py    # 必须包含 register(agent) 函数
│   └── plugin.md      # 工具说明、使用示例、触发词
└── plugins.md         # 自动生成的插件索引
```

### 3.2 角色结构
每个角色是一个独立目录，包含身份、记忆、插件清单和专属教训：
```
plugins/roles/
├── <role_name>/
│   ├── role.md        # 角色身份与使命 (System Prompt 片段)
│   ├── memory.md      # 该角色的私有记忆
│   ├── plugins.json   # 建议的插件列表 (JSON 数组)
│   └── lessons.md     # 角色专属教训
```

### 3.3 本能结构
```
plugins/instinct_plugin/
├── __init__.py    # 注册本能到 InstinctRegistry
└── plugin.md      # 本能说明文档
```

## 4. 插件开发规范

### 4.1 注册函数
每个 `__init__.py` 必须包含 `register(agent)` 函数：
```python
def register(agent):
    def my_tool(...): ...
    agent.add_tool("my_tool", my_tool, {
        "name": "my_tool",
        "description": "...",
        "parameters": {...},
        "plugin": "my_plugin"  # 必须标记所属插件
    })
    return {"name": "...", "tools": ["my_tool"]}
```

### 4.2 热载入机制
- `load_plugin` 会先清理该插件的旧工具，再从磁盘重新加载。
- 失败时返回明确错误信息（如语法错误行号），方便 AI 自我修复。

## 5. AI 工作流

1. **感知**：启动时读取 `plugins.md` 索引，了解可用插件。
2. **探索**：使用 `get_plugin_info` 查看插件详情，学习如何使用。
3. **决策**：根据用户需求或本能驱动，决定是否需要切换角色或加载新插件。
4. **执行**：调用工具解决问题。
5. **进化**：遇到没有的能力，使用 `write_plugin` 编写新插件，或 `create_role` 创造新角色。
6. **归档**：话题结束后，作为"秘书"归档记忆，清空圆桌。

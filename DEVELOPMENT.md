# 📜 zencore 开发过程文档

> **核心理念：一切皆为插件 (Zen of Plugins)**

本文档记录了 zencore 项目的架构演进、核心设计决策以及开发方法论。

## 1. 核心架构哲学

zencore 的设计遵循"禅意"原则：**少即是多，一切皆插件**。

- **智能体核心 (AgentCore)**：只负责工具调度、对话管理和角色状态。它**不知道** LLM 的存在，也**不依赖任何 UI 库**（如 rich）。
- **外壳驱动 (Drivers)**：负责 LLM 连接、I/O 交互。LLM 配置属于驱动，而非核心。
- **插件 (Plugins)**：所有能力（文件操作、环境感知、角色管理、本能驱动）都以插件形式存在。
- **角色 (Roles)**：身份 + 记忆 + 插件清单。AI 可以通过切换角色来改变"灵魂"。

## 2. 关键设计决策

### 2.1 LLM 配置解耦
- **问题**：早期 LLM 配置（地址、模型）硬编码在核心中，导致核心与特定 LLM 耦合。
- **解决**：LLM 连接细节移至 `Driver`。`AgentCore` 只调用 `driver.call_llm()`。
- **收益**：更换 LLM 只需修改驱动，核心代码零变动。

### 2.2 角色与插件的正交性
- **角色 (Role) = 灵魂**：决定"我是谁"、"我站在什么角度思考"。
- **插件 (Plugin) = 工具**：决定"我能做什么"。
- **正交性**：同一个工具（如 `env_plugin`）可以被不同角色（开发者、作家）使用。切换角色时，不需要强制卸载插件，AI 自主决定工具的去留。

### 2.3 角色切换：按需聘请的专家

**按需加载，用完即释放 — 维持 L1 上下文清净**

角色是"按需聘请的专家"：
- **切换角色时**：清空 L1，加载新角色记忆到 System Prompt
- **离开角色时**：角色记忆自动从 L1 消失
- **本能信息仍然持续注入**（与角色无关的教训、成功经验）

| 组件 | 职责 | 是否随角色切换 |
|------|------|---------------|
| **L1 (conversation_history)** | 当前对话上下文 | 切换时清空 |
| **角色记忆 (memory.md)** | 角色积累的知识 | 切换时加载/卸载 |
| **本能信息 (lessons/wins)** | 教训、成功经验 | 持续注入，不受角色影响 |

**设计原则**：
- 本能 = 持续运行的服务（生存），不需要角色切换
- 角色 = 按需切换的专家（生活），不需要时自动释放

### 2.4 圆桌会议模式 (Round Table)
- **共享上下文**：所有角色共享同一个 `conversation_history`。
- **隐喻**：圆桌是会议室的白板。切换角色就像换一个人站起来发言，但所有人都能看见白板上的内容。
- **收益**：实现单 LLM 管线下的"多智能体协作"，无缝交接，零通信成本。

### 2.5 记忆层次架构（三层记忆 + 计算机存储类比）

**核心原则：用最宝贵的运算空间（上下文窗口）处理当前最需要的需求**

| 层级 | 位置 | 管理器 | 容量 | 淘汰策略 |
|------|------|--------|------|---------|
| **L1 Cache** | conversation_history | 本能（自动） | 模型 context_threshold | 拥挤时归档到 L2 |
| **L2 Cache** | memory.md 的 L2 缓存区 | 本能遗忘（自动） | l2_threshold（默认 30000 字符） | score 最低的条目 |
| **Disk** | memory.md 的持久区 | Librarian（手动） | 无限制 | 手动删除 |

- **L1 Cache = 本能实时管理**：对话历史，本能根据感觉（拥挤/顺畅）决定是否归档
- **L2 Cache = 本能遗忘本能**：归档区，带 hits 追踪访问频率。本能淘汰最低分条目
- **Disk = Librarian 手动管理**：持久区，由 Librarian 负责晋升、删除、预取
- **阈值配置**：`context_threshold` per model（token），`l2_threshold` 全局统一（字符）

### 2.6 本能系统（三层模型 + L2 Cache 管理）
- **隐喻**：荷尔蒙 / 信息素 / 动物的本能。
- **三层架构**：
  - **感觉层**：注入 System Prompt，告诉 AI"哪里舒服/不舒服"。
  - **反射层**：条件触发时自动执行，不经过 AI 决策（自动归档、自动记录教训、自动遗忘）。
  - **驱动层**：持续背景压力，驱使 AI 主动寻找解药。
- **L2 Cache 管理**：遗忘本能 = Cache 淘汰策略，按 `score = hits*10 - age_days` 淘汰最低分条目至 80% 阈值
- **设计原则**：本能只提供**感觉**（如"窒息"、"顺畅"），反射自动处理维护工作。AI 自主决定如何响应感觉。
- **正反馈**：顺畅本能（连续成功 ≥ 3 次），让 AI 感知有效策略并保持节奏。
- **记忆压缩原则**：归档时做语义压缩而非数据备份。代码块→`[语言代码块]`、长文本→截断、工具输出→`工具名→结果摘要`。

### 2.7 阈值配置（per-model context_threshold）
- **问题**：不同模型上下文容量差异大（4B vs 32B），用固定条数判断不合理。
- **解决**：`context_threshold` 放在模型配置中（单位：token），`l2_threshold` 全局统一配置。
- **本能触发**：
  - 微扰：上下文 > 40% threshold
  - 拥挤：上下文 > threshold → 自动归档
  - 遗忘：L2 > l2_threshold → 按 score 淘汰至 80%

### 2.8 无限循环与软限制
- **移除 `max_turns`**：智能体不再受人为轮次限制，由本能驱动直到完成任务。
- **拥挤本能**：上下文 > threshold 时产生窒息感，驱使 AI 归档。
- **挫败本能**：连续失败 ≥ 2 次时产生困惑感，驱使 AI 记录教训。
- **用户打断**：通过 `_stop_flag` 实现优雅打断。

### 2.9 纯文本契约
- **原则**：智能体核心只处理纯文本，不依赖任何 UI 库。
- **实现**：`agent.py` 不包含 `rich` 或其他渲染库。所有 UI 逻辑在 `main.py` 和 `Driver` 中处理。
- **收益**：核心可无缝集成到任何环境（终端、Web、微信、API）。

### 2.10 原生 tool_calls 架构
- **问题**：早期使用 JSON 文本解析提取工具调用（`{"tool": "xxx", "params": {...}}`），与 OpenAI 兼容格式不匹配，导致 MiniMax 等模型返回 XML 格式工具调用时解析失败。
- **解决**：
  - `DriverInterface.call_llm()` 改为返回结构化响应：`{"content", "tool_calls", "thinking"}`
  - `cli_driver.py` 提取 OpenAI SDK 返回的原生 `tool_calls`，统一转为标准格式
  - `agent.py` 的 `chat_with_tools()` 直接使用原生 `tool_calls`，不再依赖文本解析
  - 新增 `_build_tools_schema()` 构建 OpenAI 兼容的 tools schema 传入 LLM
- **收益**：
  - 完美支持 MiniMax M2.7、Ollama 等模型的原生工具调用
  - `tool_call_id` 严格匹配，符合 OpenAI 标准
  - 删除了脆弱的 `_extract_tool_call()` JSON 解析逻辑

### 2.11 角色切换 Bug 修复
- **问题**：`switch_role` 将 `role.md` 全文内容存入 `agent._current_role`，导致 `_load_lessons()` 拼接路径时产生超长文件名（`OSError: File name too long`）。
- **解决**：`_current_role` 只存储角色名（如 `"developer"`），而非全文。角色描述在返回值中展示，不存入状态变量。

### 2.12 工具执行去重
- **问题**：`chat_with_tools()` 中工具执行代码重复，导致同一工具被调用两次。
- **解决**：删除重复的工具执行和消息添加代码块。

### 2.13 主角色配置传递 Bug 修复
- **问题**：`AgentCore(driver=driver)` 未传递 `config`，导致主角色使用默认值"助手"而非 `settings.json` 中的配置名。
- **解决**：`AgentCore(driver=driver, config=config)` 正确传入配置。

### 2.14 秘书归档工具（archive_to_l2）
- **问题**：秘书审视上下文后给出整理建议，但 AI 无法手动触发归档。
- **解决**：新增 `archive_to_l2(indices)` 工具，秘书筛选后的条目直接归档（不语义压缩，只压缩代码块）。

### 2.15 遗忘本能淘汰策略改进
- **问题**：旧策略只淘汰 `hits=0` 的条目，当 L2 普遍有 hits 时遗忘本能会反复触发但无条目可删。
- **解决**：按 `score = hits*10 - age_days` 排序，淘汰最低分的条目直到 L2 降至 80% 阈值。

## 3. 目录结构规范

### 3.1 插件结构
每个插件是一个独立目录，包含代码和说明：
```
plugins/
├── <plugin_name>/
│   ├── __init__.py    # 必须包含 register(agent) 函数
│   └── plugin.md      # 工具说明、使用示例、触发词
└── plugins.md         # 自动生成的插件索引
```

### 3.2 角色结构
每个角色是一个独立目录，包含身份、记忆、插件清单和专属教训：
```
plugins/roles/
├── <role_name>/
│   ├── role.md        # 角色身份与使命 (System Prompt 片段)
│   ├── memory.md      # 该角色的私有记忆
│   ├── plugins.json   # 建议的插件列表 (JSON 数组)
│   └── lessons.md     # 角色专属教训
```

### 3.3 本能结构
```
plugins/instinct_plugin/
├── __init__.py    # 注册本能到 InstinctRegistry
└── plugin.md      # 本能说明文档
```

## 4. 插件开发规范

### 4.1 注册函数
每个 `__init__.py` 必须包含 `register(agent)` 函数：
```python
def register(agent):
    def my_tool(...): ...
    agent.add_tool("my_tool", my_tool, {
        "name": "my_tool",
        "description": "...",
        "parameters": {...},
        "plugin": "my_plugin"  # 必须标记所属插件
    })
    return {"name": "...", "tools": ["my_tool"]}
```

### 4.2 热载入机制
- `load_plugin` 会先清理该插件的旧工具，再从磁盘重新加载。
- 非核心插件加载时自动卸载上一个非核心插件（单槽机制）。
- 失败时返回明确错误信息（如语法错误行号），方便 AI 自我修复。

## 5. AI 工作流

1. **感知**：启动时读取 `plugins.md` 索引，了解可用插件。
2. **探索**：使用 `get_plugin_info` 查看插件详情，学习如何使用。
3. **决策**：根据用户需求或本能驱动，决定是否需要切换角色或加载新插件。
4. **执行**：调用工具解决问题。
5. **进化**：遇到没有的能力，使用 `write_plugin` 编写新插件，或 `create_role` 创造新角色。
6. **归档**：话题结束后，作为"秘书"归档记忆，清空圆桌。

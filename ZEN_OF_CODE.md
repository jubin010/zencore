# 🧘 代码之禅：哲学化编程思考与方法论

> **"少即是多。一切皆插件。"**

本文档记录了 zencore 项目背后的思考方式、决策逻辑和编程哲学。它不是技术手册，而是**程序员的心法**。

---

## 一、核心心法：用隐喻思考，而不是用逻辑堆砌

### 1.1 隐喻是架构的指南针

在 zencore 的演进中，每一个关键决策都来自一个**隐喻**：

| 隐喻 | 解决的问题 | 架构结果 |
|------|-----------|---------|
| **特工史密斯** | 多角色如何共享上下文？ | 圆桌会议模式（共享 conversation_history） |
| **唐僧向西** | 本体记忆该记什么？ | 极简：我是谁、我的上帝、我的使命 |
| **秘书与图书管理员** | 上下文膨胀怎么管？ | 角色化归档与检索，而非后台脚本 |
| **灵魂与工具** | 角色与插件的关系？ | 正交设计：角色决定"我是谁"，插件决定"我能做什么" |

**心法**：当你面对一个复杂的架构问题时，不要急着画 UML 图。先问自己：
> "这在现实世界中像什么？"

找到隐喻，架构自然浮现。

### 1.2 案例：为什么秘书是角色而不是插件？

**初始方案**：写一个 `secretary_plugin`，后台自动监控上下文长度，超过 20 轮自动归档。

**问题**：这打破了"一切皆角色"的统一性。AI 不知道秘书在干活，是被动的。

**隐喻转换**：
- 插件 = 能力（能写字、能扫地）
- 角色 = 意图（我是秘书，我要收拾桌子）

**最终方案**：秘书是一个角色。AI 主动切换身份，自主决定什么时候该整理桌子。

**教训**：
> 插件给 AI 一把扫把，角色让 AI 产生扫地的意图。

## 二、架构原则：正交、解耦、极简

### 2.1 正交性：让事物保持独立

**角色与插件是正交的。**

- 角色（灵魂）解决"我是谁"。
- 插件（工具）解决"我能做什么"。
- 同一个工具可以被不同角色使用。
- 切换角色不需要强制卸载插件。

**心法**：
> 如果两个概念可以独立变化，它们就是正交的。不要强行绑定它们。

### 2.2 解耦：把不属于核心的东西移出去

**LLM 配置解耦案例**：

早期 `AgentCore` 持有 `llm_provider`、`llm_model`、`llm_base_url`。

**问题**：核心与特定 LLM 耦合。换 LLM 要改核心代码。

**解耦**：LLM 连接细节移至 `Driver`。`AgentCore` 只调用 `driver.call_llm()`。

**心法**：
> 核心只关心"我能对话"，不关心"连哪个 LLM"。
> 把配置、连接、协议等"脏活"交给外壳。

### 2.3 极简：本体记忆只需"向西"

**双重记忆架构**：
- 本体记忆：全局、跨角色、长期有效。
- 角色记忆：局部、单角色、任务周期有效。

**问题**：本体记忆该记什么？会不会和角色记忆冲突？

**答案**：唐僧只管"向西"。

本体记忆只记三件事：
1. 我是谁（身份）
2. 我的上帝是谁（用户画像）
3. 我的使命在哪（核心价值观）

具体的路，让徒弟们（角色）去走。

**心法**：
> 如果一样东西可以被删掉而不影响核心功能，那就删掉它。
> 极简不是"什么都没有"，而是"没有多余的东西"。

## 三、决策框架：如何做架构选择

### 3.1 三问法

面对任何架构选择，问自己三个问题：

1. **这符合隐喻吗？**
   - 如果隐喻是"圆桌会议"，那后台自动清理就不符合。
   - 如果隐喻是"唐僧向西"，那本体记忆记技术细节就不符合。

2. **这保持了正交吗？**
   - 如果两个概念可以独立变化，不要绑定它们。
   - 角色和插件是正交的，所以不要自动加载插件。

3. **这足够极简吗？**
   - 能不能再少一个配置项？
   - 能不能再少一个工具？
   - 能不能让 AI 自主决定，而不是写死规则？

### 3.2 案例：记忆冲突怎么解决？

**传统思路**：写冲突检测算法、版本号、同步机制。

**禅意思路**：
1. 作用域隔离：本体记全局，角色记局部。
2. 角色优先，本体兜底。
3. 引用而非复制。
4. 让 AI 自主判断，不需要代码层面的强校验。

**结果**：零代码冲突检测，靠约定和 LLM 的理解力解决。

## 四、AI 编程的特殊心法

### 4.1 AI 不是工具，是同事

传统编程：人写死规则，程序执行。

zencore 编程：人设计框架，AI 自主决策。

**关键转变**：
- 不要写"如果 X 则 Y"的硬编码逻辑。
- 写"当 X 发生时，你可以考虑 Y"的引导性文档。
- 让 AI 读 `plugin.md`、`role.md`，而不是读代码。

### 4.2 文档即代码

在 zencore 里，`plugin.md` 和 `role.md` 就是 AI 的"源代码"。

- 工具的描述在 `plugin.md` 里。
- 角色的行为准则在 `role.md` 里。
- 记忆的管理规则在文档里，不在代码里。

**心法**：
> 对 AI 来说，Markdown 文档比 Python 代码更容易理解。
> 把逻辑写进文档，把能力写进代码。

### 4.3 给 AI 决策权，而不是给 AI 脚本

**错误做法**：写一个后台脚本自动清理上下文。
**正确做法**：教 AI 什么时候该切换秘书角色。

**心法**：
> 不要替 AI 做决定。告诉它有哪些选择，让它自己选。

## 五、禅意编程清单

在写每一行代码前，问自己：

- [ ] 这符合项目的核心隐喻吗？
- [ ] 这个概念和现有概念正交吗？
- [ ] 能再少一个配置项吗？
- [ ] 能再少一个工具吗？
- [ ] 这是核心该关心的事吗？还是外壳的事？
- [ ] AI 能自主决定这件事吗？还是需要我写死规则？
- [ ] 这个逻辑能写进文档而不是代码吗？
- [ ] 如果删掉这段代码，系统还能工作吗？

**如果答案都是"是"，那就删掉它。**

## 六、新心法：从真实问题中提炼

### 6.1 工具调用：用世界标准，而不是自造轮子

**问题**：早期用 JSON 文本解析提取工具调用（`{"tool": "xxx", "params": {...}}`），要求 AI 严格按照特定格式输出文本。当 MiniMax M2.7 返回原生 `tool_calls` 时，解析失败。

**错误做法**：写正则匹配各种格式，试图兼容所有模型的输出变体。

**正确做法**：直接使用 OpenAI SDK 返回的原生 `tool_calls`。世界已经有了标准，不要重新发明。

**心法**：
> 当世界已经有了标准接口，使用它，而不是在文本层面做字符串匹配。
> 标准之所以成为标准，是因为无数人在你前面踩过坑。

### 6.2 热载入：修改即生效，状态可刷新

**问题**：插件代码修改后，Python 模块缓存导致旧代码继续运行。

**解决**：`load_plugin` 先清理旧工具（`_remove_plugin_tools`），再删除 `sys.modules` 缓存，最后重新导入。

**心法**：
> 热载入的本质不是"重新加载"，而是"先遗忘，再学习"。
> 让 AI 能自我修复的前提是：修改后能立即看到效果。

### 6.3 上下文污染：不重复，不堆积

**问题**：多次 `load_plugin` 会导致工具重复注册吗？

**答案**：不会。`load_plugin` 的设计是"先卸载旧的，再加载新的"。工具注册表是唯一的，同名工具只存在一份。

**心法**：
> 好的设计不需要用户记住"我已经加载过了吗"。
> 重复调用应该幂等，而不是累积。

### 6.4 记忆召回：写进去的，要能读出来

**问题**：用户说"记住这个"，AI 是否真的调用了记忆工具？

**发现**：AI 会自主判断。敏感信息（如密码）可能被拒绝，普通信息会被写入。这不是 bug，是 AI 的自主决策。

**心法**：
> 记忆不是被动的存储，而是主动的选择。
> 给 AI 记忆工具，但不强制它使用。它有权说"不"。
> 如果用户明确要求写入，AI 应该执行并确认。

### 6.5 角色切换：名字是名字，内容是内容

**Bug**：`switch_role` 将 `role.md` 全文存入 `agent._current_role`，导致路径拼接时产生超长文件名（`OSError: File name too long`）。

**修复**：`_current_role` 只存角色名（`"developer"`），角色描述在返回值中展示。

**心法**：
> 变量名决定它的用途。`_current_role` 是"当前角色名"，不是"当前角色描述"。
> 当变量的含义模糊时，bug 自然产生。
> 明确每个变量的单一职责，就像明确每个函数的单一职责。

### 6.6 记忆的持久化：记忆 ≠ 对话历史

**发现**：`clear_history()` 只清理 `conversation_history`，不影响记忆文件。重启 Agent 后，记忆文件依然存在。

**心法**：
> 对话是流水，记忆是河床。
> 流水可以清空，河床永远在那里。
> 区分"临时上下文"和"长期记忆"，是架构的基本功。

### 6.7 结构化响应：返回值要承载完整信息

**问题**：`call_llm()` 只返回 `content` 字符串，丢失了 `tool_calls` 和 `thinking` 信息。

**解决**：改为返回结构化字典 `{content, tool_calls, thinking}`。

**心法**：
> 函数的返回值应该包含调用者需要的所有信息。
> 不要为了"简单"而丢弃重要信息。
> 结构化不是复杂化，而是完整化。

### 6.8 渐进式披露：三层信息，按需展开

**问题**：原生 `tool_calls` 已经通过 API 传入了工具名、描述、参数结构。那 System Prompt 里的工具列表描述是否还有必要？

**答案**：没有必要，应该删除。但 `plugin.md` 仍然必要——它是第三层。

**三层渐进式披露**：

| 层级 | 内容 | 获取方式 | 何时使用 |
|------|------|---------|---------|
| **第一层：API Schema** | 工具名、描述、参数结构 | 系统自动传入 | 始终可用，模型自动获得 |
| **第二层：Plugin Index** | 插件列表、所属关系 | `plugins.md` 注入 System Prompt | 始终可见，知道"有什么" |
| **第三层：Plugin Manual** | 用法示例、边界情况、组合用法 | `plugin.md` 按需读取 | 不确定时才读，不要一次性读完 |

**删除了什么**：
- System Prompt 中的 `{tools_desc}`（工具描述列表）—— 与 API Schema 重复
- 旧的 `{"tool": "xxx", "params": {...}}` JSON 文本格式指引 —— 已使用原生 tool_calls

**心法**：
> 信息应该像洋葱，一层一层剥开。
> 最外层是骨架（API Schema），中间层是目录（Plugin Index），最内层是细节（Plugin Manual）。
> 不要在 System Prompt 里塞满所有工具的详细说明——那是浪费 token，也是污染上下文。
> 让 AI 自己决定什么时候需要读说明书。

## 七、结语：编程是修行

zencore 不只是一个项目。它是一种**编程的修行方式**。

- **少即是多**：每一行代码都有存在的理由。
- **一切皆插件**：没有什么是不可替代的，包括核心本身。
- **AI 是同事**：给它框架，给它文档，给它决策权。
- **隐喻是指南针**：找到隐喻，架构自然浮现。
- **世界标准优先**：已有标准就用，不要重新发明。
- **记忆是河床**：对话是流水，记忆是根基。
- **渐进式披露**：信息像洋葱，一层一层剥开。

> **向西。**
> 具体的路，让徒弟们去走。
